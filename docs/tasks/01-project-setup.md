# Stage 1. Project Setup

## Goal

Поднять рабочую основу SPA на `Vue 3 + Vite + TypeScript`, чтобы дальнейшие фичи добавлялись поверх уже собранной архитектуры, а не в хаотичный каркас.

## Context

Из `PROJECT_STRUCTURE.md` важно сразу придерживаться рекомендованного разбиения на `app/`, `pages/`, `components/`, `stores/`, `types/`, `composables/`, `constants/`, `utils/`.

Из `STORE_SCHEMA.md` следует учесть, что `settingsStore` должен быть доступен рано, потому что влияет на `locale`, `currency` и `theme` на уровне всего приложения.

## What To Build

- Базовый проект на `Vite + Vue 3 + TypeScript`.
- Подключенные зависимости: `pinia`, `vue-router`, `vue-i18n`, `@vueuse/core`.
- Точка входа `main.ts` с инициализацией приложения, роутера, стора и i18n.
- Базовая файловая структура в `src/`.
- Роуты для страниц `Dashboard`, `Transactions`, `Analytics`, `Settings`.
- Базовый layout приложения с навигацией и контейнером страницы.
- Настроенные алиасы, если они упрощают импорты и уже поддерживаются проектом.

## Detailed Tasks

1. Проверить текущую конфигурацию проекта и убедиться, что стек действительно `Vue 3 + Vite + TypeScript`.
2. Установить и подключить зависимости из MVP-стека: `pinia`, `vue-router`, `vue-i18n`, `@vueuse/core`.
3. Создать структуру каталогов по образцу из `PROJECT_STRUCTURE.md`:
   - `src/app/i18n`
   - `src/app/router`
   - `src/app/providers`
   - `src/components/{analytics,dashboard,layout,settings,transactions,ui}`
   - `src/pages`
   - `src/stores`
   - `src/types`
   - `src/composables`
   - `src/constants`
   - `src/utils`
4. Подготовить `src/app/router/index.ts` и при необходимости `routes.ts`, чтобы route-level код не размазывался по компонентам.
5. Завести страницы:
   - `DashboardPage.vue`
   - `TransactionsPage.vue`
   - `AnalyticsPage.vue`
   - `SettingsPage.vue`
6. Настроить `main.ts`:
   - создать приложение
   - подключить `Pinia`
   - подключить `Vue Router`
   - подключить `i18n`
   - смонтировать приложение
7. Сделать базовый layout уровня приложения:
   - `AppShell`
   - `AppNav`
   - контейнер для контента страницы
8. Убедиться, что layout не содержит доменной логики и используется как каркас для route pages.
9. Настроить алиасы, если проект использует `@` или аналогичный короткий путь для `src/`.
10. Проверить, что переходы между страницами работают, а пустые страницы отображаются без ошибок.

## Dependencies

- Этот этап должен быть завершен до реализации stores, CRUD транзакций и аналитики.
- Структура каталогов должна быть готова до массового добавления компонентов и composables.
- `settingsStore` можно пока не реализовывать полностью, но архитектура должна допускать его раннее подключение.

## Acceptance Criteria

- Приложение запускается локально без ошибок.
- Все основные зависимости подключены и реально используются приложением.
- В `src/` присутствует согласованная структура, близкая к `PROJECT_STRUCTURE.md`.
- `main.ts` подключает router, `Pinia` и `i18n`.
- Доступны маршруты `/`, `/transactions`, `/analytics`, `/settings`.
- Есть базовый layout с навигацией между страницами.
- Страницы открываются и рендерятся внутри общего каркаса.

## Notes

- Не стоит перегружать этот этап бизнес-логикой: здесь нужен именно каркас.
- Если часть структуры еще не используется, достаточно создать каталоги и минимальные заглушки, но без лишнего boilerplate.
- Route-level код держать в `pages/`, а не в feature-компонентах.
