# Stage 3. Transactions Feature

## Goal

Реализовать основной пользовательский сценарий приложения: просмотр, создание, редактирование и удаление транзакций с корректной формой и опорой на доменные stores.

## Context

Из `PROJECT_STRUCTURE.md` следует, что страница должна собираться в `pages/`, визуальные блоки жить в `components/transactions/`, а доменная логика оставаться в store и shared-утилитах.

Из `STORE_SCHEMA.md` транзакции являются главным store проекта, поэтому UI этой фичи должен опираться на `useTransactionsStore` и `useCategoriesStore` как на source of truth.

## What To Build

- Полноценную страницу `/transactions`.
- Компоненты для списка транзакций, элемента списка, формы и модального окна.
- CRUD-операции над транзакциями.
- Базовую валидацию пользовательского ввода.
- Корректное состояние страницы при отсутствии данных.

## Detailed Tasks

1. Создать маршрутную страницу `TransactionsPage.vue`, которая собирает экран из feature-компонентов.
2. Реализовать `TransactionList` для вывода отсортированных транзакций.
3. Реализовать `TransactionItem` с отображением:
   - типа операции
   - суммы
   - категории
   - даты
   - заметки
4. Сделать `TransactionForm` с полями:
   - `type`
   - `amount`
   - `categoryId`
   - `date`
   - `note`
5. Добавить `TransactionModal` или другой согласованный контейнер открытия формы.
6. Подключить создание транзакции через `transactionsStore.addTransaction(payload)`.
7. Подключить редактирование через `transactionsStore.updateTransaction(id, payload)`.
8. Подключить удаление через `transactionsStore.removeTransaction(id)`.
9. Добавить валидацию формы:
   - сумма больше нуля
   - тип обязателен
   - категория обязательна
   - дата обязательна
10. Учесть, что список категорий должен зависеть от выбранного `type` транзакции.
11. Реализовать сортировку по дате, приоритетно через getter/store-level логику, а не прямо в шаблоне компонента.
12. Сделать empty state для случая, когда транзакций еще нет.
13. Обеспечить удобный сценарий открытия формы как для создания, так и для редактирования.
14. Проверить, что изменения формы после сохранения сразу отражаются в списке благодаря реактивности store.

## Dependencies

- Требует реализованных типов и stores из Stage 2.
- Использует категории как обязательный справочник для выбора `categoryId`.
- Становится базой для dashboard, analytics, filters и budgets.

## Acceptance Criteria

- Страница `/transactions` отображает список транзакций.
- Пользователь может создать транзакцию с валидными данными.
- Пользователь может отредактировать существующую транзакцию.
- Пользователь может удалить транзакцию.
- Список отсортирован по дате в предсказуемом порядке.
- При отсутствии данных показывается понятное empty state.
- Форма не сохраняет невалидные значения.

## Notes

- Не стоит хранить временные ошибки формы и состояние каждого инпута в store.
- Если модальное состояние локально только для страницы транзакций, его можно оставить на уровне страницы или composable, а не выносить в отдельный UI store.
- Если в проекте появится отдельная UI-модальная инфраструктура, эту фичу можно адаптировать позже без переписывания доменной логики.
