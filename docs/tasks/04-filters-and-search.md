# Stage 4. Filters And Search

## Goal

Добавить управляемую фильтрацию и поиск по транзакциям, чтобы список был пригоден для реального использования и поддерживал deep-linking через query params.

## Context

Из `PROJECT_STRUCTURE.md` и `STORE_SCHEMA.md` важно помнить, что глобальные фильтры не обязательно хранить в store. Если они относятся только к экрану списка, им лучше жить в `pages/`, `components/transactions/` или отдельном `composable`.

## What To Build

- UI фильтров для списка транзакций.
- Реактивную систему фильтрации с понятной логикой комбинирования условий.
- Синхронизацию фильтров с URL query params.
- Возможность быстро сбросить фильтры к дефолтному состоянию.

## Detailed Tasks

1. Создать `TransactionFilters` как отдельный feature-компонент.
2. Определить локальную модель фильтров:
   - `type`
   - `categoryId`
   - `dateFrom`
   - `dateTo`
   - `search`
3. Реализовать фильтрацию списка по типу операции `income/expense`.
4. Реализовать фильтрацию по категории.
5. Реализовать фильтрацию по диапазону дат.
6. Реализовать поиск по `note`.
7. Определить порядок применения фильтров, чтобы он был стабилен и легко тестировался.
8. Вынести повторяемую логику в `useTransactionFilters`, если фильтры начинают разрастаться.
9. Синхронизировать состояние фильтров с query params роутера:
   - загрузка фильтров из URL при открытии страницы
   - обновление URL при изменении фильтров
10. Сделать кнопку `Reset filters`, которая очищает UI-состояние и query params.
11. Проверить поведение при частично заполненных фильтрах, например только `dateFrom` без `dateTo`.
12. Убедиться, что фильтрация не ломает сортировку списка и корректно работает вместе с CRUD.

## Dependencies

- Требует готовой страницы транзакций и базового CRUD из Stage 3.
- Использует router из Stage 1 и stores из Stage 2.
- Логика фильтрации должна быть совместима с будущими тестами из Stage 12.

## Acceptance Criteria

- Пользователь может фильтровать транзакции по типу, категории и диапазону дат.
- Поиск по заметке работает и комбинируется с остальными фильтрами.
- Состояние фильтров отражается в URL query params.
- При открытии страницы по ссылке фильтры восстанавливаются из URL.
- Кнопка сброса полностью очищает фильтры и URL-параметры.
- Список после фильтрации остается предсказуемым и не ломает основной UX.

## Notes

- Не нужно преждевременно делать глобальный `filtersStore`, если фильтры используются только на одной странице.
- Логику сравнения дат и строкового поиска лучше держать вне шаблона компонента.
- Если поиск по заметке должен быть нечувствителен к регистру, это стоит зафиксировать сразу.
