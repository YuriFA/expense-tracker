# Stage 9. Budgets

## Goal

Добавить контроль месячных лимитов по категориям, чтобы приложение не только учитывало траты, но и помогало пользователю отслеживать перерасход.

## Context

Из `STORE_SCHEMA.md` MVP ограничивается периодом `'month'`, а `budgetsStore` зависит от `transactionsStore` для расчета фактических расходов по категориям.

## What To Build

- UI для задания месячного лимита по категории.
- Отображение фактических расходов относительно лимита.
- Состояния `normal`, `warning`, `exceeded`.
- Общую сводку по бюджетам.

## Detailed Tasks

1. Реализовать UI для создания или изменения месячного бюджета по категории.
2. Использовать `useBudgetsStore.setBudget(categoryId, limit)` как основную точку записи.
3. Ограничить период бюджетов MVP-версией `'month'`, без недельных и кастомных режимов.
4. Посчитать фактические расходы по каждой категории за текущий месяц через данные `transactionsStore`.
5. Сопоставить фактическую сумму с лимитом.
6. Реализовать прогресс-отображение по каждой категории.
7. Добавить визуальную подсветку приближения к лимиту, например после заданного порога.
8. Добавить отдельное состояние превышения бюджета.
9. Сделать summary по всем бюджетам:
   - сколько категорий в норме
   - сколько близко к лимиту
   - сколько превышено
10. Определить, где показывать бюджеты:
   - отдельный блок на dashboard
   - отдельная секция или компонент
   - страница настроек или аналитики, если это лучше укладывается в UX
11. Проверить поведение, если для категории нет бюджета.
12. Проверить поведение, если бюджет есть, но по категории еще нет расходов.

## Dependencies

- Требует готовых categories и transactions stores.
- Использует `budgetsStore` из Stage 2.
- Пересекается с dashboard и analytics, если бюджеты нужно показывать в сводных экранах.

## Acceptance Criteria

- Пользователь может задать месячный бюджет по категории.
- Приложение показывает, сколько уже потрачено относительно лимита.
- Есть отдельная индикация приближения к лимиту и превышения.
- Summary по бюджетам рассчитывается корректно.
- Логика работает только для расходных категорий или это ограничение явно описано в UX.

## Notes

- Бюджеты логично применять к `expense`-категориям, а не ко всем категориям подряд.
- Порог предупреждения лучше зафиксировать явно, чтобы потом его можно было легко менять.
- Если формат summary понадобится в нескольких местах, вычисления стоит вынести в `utils` или composable.
