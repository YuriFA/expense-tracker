# Stage 10. Import/Export

## Goal

Дать пользователю безопасный способ сохранить свои данные вне приложения и восстановить их обратно без риска поломать хранилище некорректным JSON.

## Context

Из `STORE_SCHEMA.md` все доменные области уже хранятся в `localStorage`, а stores содержат методы `replaceAll(...)`, которые удобно использовать при массовой замене данных.

Из `PROJECT_STRUCTURE.md` нормализацию импортируемых данных лучше держать в `utils/`, а UI импорт/экспорт компонентов можно положить в `components/settings/`.

## What To Build

- Экспорт текущих пользовательских данных в JSON-файл.
- Импорт JSON-файла обратно в приложение.
- Базовую валидацию и нормализацию структуры.
- Защиту от частично сломанных или несовместимых данных.

## Detailed Tasks

1. Определить экспортируемую структуру данных:
   - `transactions`
   - `categories`
   - `budgets`
   - `settings`
2. Реализовать `Export`-сценарий:
   - собрать актуальные данные из stores
   - сериализовать в JSON
   - инициировать скачивание файла
3. Реализовать `Import`-сценарий:
   - выбрать JSON-файл
   - прочитать содержимое
   - распарсить JSON
4. Добавить базовую валидацию верхнеуровневой структуры.
5. Добавить нормализацию данных в `utils/`:
   - привести отсутствующие поля к дефолтным значениям, если это допустимо
   - отбросить явно некорректные записи или отклонить импорт целиком
6. Определить стратегию импорта:
   - полностью заменить текущие данные
   - или объединять с существующими, если есть явное требование
7. Для MVP зафиксировать один понятный вариант, предпочтительно `replace all`.
8. Использовать store actions вроде `replaceAll(...)` и `hydrate(...)`, а не мутировать state в обход store API.
9. Добавить UI-компонент `ImportExportData`.
10. Показать пользователю понятный результат импорта:
   - успех
   - ошибка структуры
   - ошибка чтения файла
11. Проверить, что после импорта данные корректно сохраняются в persistence и отображаются на экранах приложения.

## Dependencies

- Требует готовых stores и persistence из Stage 2.
- Пересекается с экраном настроек из Stage 8.
- Желательно иметь завершенный CRUD, чтобы импортированные данные можно было сразу проверить в интерфейсе.

## Acceptance Criteria

- Пользователь может экспортировать все свои данные в JSON.
- Пользователь может импортировать валидный JSON обратно.
- Некорректный JSON не ломает приложение и не портит текущее состояние silently.
- Импорт использует store API, а не прямую запись в state.
- После успешного импорта данные видны в приложении и переживают перезагрузку.

## Notes

- Для MVP лучше сделать понятный и строгий формат JSON, чем пытаться поддержать много вариантов структуры.
- Если импорт заменяет все данные, это нужно явно показать в UI до подтверждения пользователя.
- Нормализация должна быть минимальной и предсказуемой, без скрытого исправления серьезно битых данных.
