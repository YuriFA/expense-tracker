# Stage 11. Polish

## Goal

Довести приложение до состояния аккуратного законченного продукта: улучшить адаптивность, empty/loading states, визуальное качество форм и общий UX первого запуска.

## Context

Из `PROJECT_STRUCTURE.md` это кросс-срезовый этап: он проходит по `layout`, `dashboard`, `transactions`, `analytics`, `settings` и `ui` компонентам.

## What To Build

- Улучшенную адаптивную оболочку приложения.
- Осмысленные empty/loading states.
- Более аккуратные формы, карточки и базовые UI-компоненты.
- Хороший сценарий первого открытия приложения без данных.

## Detailed Tasks

1. Проверить layout на мобильных ширинах:
   - навигация
   - отступы
   - поведение карточек
   - ширину форм и модалок
2. Исправить проблемные места в адаптивности на основных экранах:
   - dashboard
   - transactions
   - analytics
   - settings
3. Добавить skeleton/loading states там, где пользователь может ожидать данные или тяжелые блоки UI.
4. Проверить, не подменяют ли skeleton states реальные empty states, где данных просто нет.
5. Добавить или улучшить empty states:
   - нет транзакций
   - нет аналитики из-за отсутствия данных
   - нет бюджетов
6. Улучшить UI форм:
   - читаемые подписи
   - понятные ошибки валидации
   - заметные primary actions
7. Улучшить UI карточек и summary-блоков:
   - контраст
   - визуальная иерархия
   - единые отступы и типографика
8. Пройти пользовательский сценарий первого запуска:
   - приложение открывается без данных
   - пользователь понимает, что делать дальше
   - есть понятная точка входа в создание первой транзакции
9. Проверить, что пустые состояния не выглядят как ошибка приложения.

## Dependencies

- Нужна готовая основная функциональность предыдущих этапов.
- Пересекается с theme support, потому что polish нужно проверять в обеих темах.
- Может потребовать доработки общих `ui` компонентов.

## Acceptance Criteria

- Приложение удобно использовать на desktop и mobile.
- Основные экраны не ломаются на узких ширинах.
- Есть осмысленные loading states там, где они нужны.
- Есть аккуратные empty states для ключевых сценариев.
- Формы и карточки выглядят цельно и читаемо.
- Первый запуск не ставит пользователя в тупик.

## Notes

- Не нужно добавлять loading state только ради факта его наличия, если данные синхронные и экран не выигрывает от этого.
- Empty state должен объяснять следующее действие, а не просто сообщать, что список пуст.
- Этот этап хорошо подходит для финального прохода по продукту глазами пользователя, а не только разработчика.
