# Stage 12. Testing

## Goal

Закрыть критические риски проекта тестами и ручной проверкой, чтобы основные пользовательские сценарии были защищены от регрессий.

## Context

Из предыдущих этапов уже ясно, что наибольшая ценность будет в проверке доменной логики, а не только рендера компонентов.

## What To Build

- Тестовую инфраструктуру на `Vitest`.
- Набор unit-тестов для ключевых store и utility-сценариев.
- Минимальный manual QA checklist для основных пользовательских потоков.

## Detailed Tasks

1. Подключить `Vitest`, если он еще не установлен и не настроен.
2. Определить структуру тестов:
   - store tests
   - utils/composables tests
   - при необходимости component tests для критичных UI-сценариев
3. Написать тесты для `transactionsStore`:
   - добавление транзакции
   - обновление транзакции
   - удаление транзакции
   - расчет totals и balance
   - сортировка транзакций
4. Написать тесты для логики фильтрации:
   - фильтр по типу
   - фильтр по категории
   - фильтр по диапазону дат
   - поиск по заметке
   - комбинация нескольких фильтров
5. Написать тесты для расчетов статистики:
   - доходы и расходы за месяц
   - расходы по категориям
   - группировка по месяцам
   - средние траты
   - определение самой затратной категории
6. Если persistence вынесен в helper, покрыть тестами и его критичное поведение.
7. Составить manual QA checklist для ключевых сценариев:
   - создание транзакции
   - редактирование транзакции
   - удаление транзакции
   - перезагрузка страницы с сохранением данных
   - фильтрация
   - смена языка
   - смена темы
   - импорт и экспорт, если фича реализована
8. Проверить, что тесты легко запускать локально одной-двумя командами.

## Dependencies

- Требует завершенной или почти завершенной основной функциональности проекта.
- Особенно зависит от stages 2, 3, 4, 5, 6, 7 и 8.
- Чем чище вынесена бизнес-логика в store и utils, тем проще и надежнее будут тесты.

## Acceptance Criteria

- В проекте есть настроенный `Vitest`.
- Основные сценарии `transactionsStore` покрыты тестами.
- Логика фильтрации покрыта тестами.
- Ключевые расчеты статистики покрыты тестами.
- Есть ручной checklist для критических пользовательских потоков.

## Notes

- На MVP нет смысла тратить много времени на хрупкие snapshot-тесты интерфейса, если не закрыта доменная логика.
- Приоритет тестов должен идти от рисков: сначала store и вычисления, потом UI.
- Если часть логики сложно тестировать, это хороший сигнал, что ее стоит упростить или лучше разделить.
